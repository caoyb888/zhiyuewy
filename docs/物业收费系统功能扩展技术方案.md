## 1. 总体设计原则

### 1.1 兼容性优先

所有新增功能在现有 `zhaoxinwy-pms` 模块体系内扩展，不破坏已有表结构。新增表以 `pms_` 为前缀，新增接口以 `/pms/v2/` 为前缀区分版本，存量接口保持不变。

### 1.2 模块归属

| 新增功能域 | 归属后端模块 | 说明 |
|---|---|---|
| 看板数据 | `zhaoxinwy-pms` | 新增 dashboard 包 |
| 企业信息 | `zhaoxinwy-system` | 复用现有用户/角色体系并扩展 |
| 基础档案 | `zhaoxinwy-pms` | 在资产管理基础上扩展 |
| 物业收费 | `zhaoxinwy-pms` | 在缴费模块基础上扩展 |
| 数据查询报表 | `zhaoxinwy-pms` + `zhaoxinwy-jmReport` | 复用积木报表服务 |
| C端微信端 | 新建 `zhaoxinwy-wxmp` 模块 | 独立微信端服务模块 |

### 1.3 技术约束

- 后端：沿用 Spring Boot 2.5.6 + MyBatis-Plus 3.4.0，不升级主框架版本
- 前端B端：沿用 Vue 2.6.12 + Element UI 2.15.6
- C端：微信小程序原生开发（uniapp 可选），通过现有微信生态 SDK 对接
- 定时任务：沿用 Quartz，新增 job 注册至现有调度中心
- 报表：新增报表通过积木报表（jmReport）配置，不单独开发

## 3. B端管理平台技术方案

### 3.1 收费模块看板

#### 3.1.1 功能描述

面向管理层提供实时数据看板，并在每天 18:00 自动推送收费情况至负责人手机（微信模板消息）。

#### 3.1.2 数据指标设计

看板聚合以下维度数据，通过独立接口分块返回，前端 ECharts 渲染：

| 指标分组 | 具体指标 | 数据来源表 |
|---|---|---|
| 收费概览 | 本月应收、实收、收费率、欠费总额 | pms_receivable, pms_payment |
| 收费方式分布 | 现金/微信/银行转账 占比 | pms_payment |
| 收缴率趋势 | 近12个月收缴率折线图 | pms_receivable, pms_payment |
| 欠费分布 | 按小区/楼栋/费用类型欠费 Top10 | pms_receivable |
| 工单数据 | 本月报修/投诉数量、处理率 | 现有工单表 |
| 业主活跃 | 微信端绑定率、本月登录次数 | pms_wx_user |

#### 3.1.3 后端实现

```java
// DashboardController.java
@RestController
@RequestMapping("/pms/v2/dashboard")
public class DashboardController {

    // 收费概览（实时查询，加 Redis 缓存 5 分钟）
    @GetMapping("/charge-overview")
    public AjaxResult chargeOverview(@RequestParam Long projectId) { ... }

    // 收费方式分布
    @GetMapping("/payment-method-dist")
    public AjaxResult paymentMethodDist(@RequestParam Long projectId,
                                         @RequestParam String month) { ... }

    // 收缴率趋势（近12个月，缓存 1 小时）
    @GetMapping("/collection-rate-trend")
    public AjaxResult collectionRateTrend(@RequestParam Long projectId) { ... }

    // 欠费 Top10
    @GetMapping("/arrears-top10")
    public AjaxResult arrearsTop10(@RequestParam Long projectId) { ... }
}
```

**缓存策略**：看板数据使用 Redis 缓存，key 格式为 `dashboard:{projectId}:{metric}:{date}`，TTL 5分钟。收费操作完成后主动清除相关缓存。

#### 3.1.4 定时推送实现

每天 18:00 由 Quartz 触发，调用微信模板消息接口，向配置了"接收推送"角色的管理员推送：

```java
        // 2. 聚合当日收费数据

```

微信模板消息内容（需在公众号后台申请模板）：

### 3.4 物业收费功能扩展

#### 3.4.1 费用定义

在现有自定义收费项基础上，扩展物业收费场景所需字段：

```sql
CREATE TABLE pms_fee_definition (
  fee_id          BIGINT PRIMARY KEY AUTO_INCREMENT,

  fee_type        TINYINT COMMENT '1周期 2临时 3押金 4预收',
  calc_type       TINYINT COMMENT '1固定金额 2按面积 3按用量 4公式',

  cycle_unit      TINYINT COMMENT '1月 2季 3年',
  overdue_type    TINYINT COMMENT '0无滞纳金 1按天 2按月',
  overdue_rate    DECIMAL(8,6) COMMENT '滞纳金比率',
  overdue_max     DECIMAL(12,2) COMMENT '滞纳金上限',
  round_type      TINYINT COMMENT '1四舍五入 2向上取整 3截尾',
  earmark_type    TINYINT DEFAULT 0 COMMENT '专款专冲：0否 1是',
```

**与现有系统衔接**：费用定义中的 `formula` 字段复用现有公式引擎（支持 `if/elsif/else/return` 语法及梯度计算），无需重新开发。

#### 3.4.4 抄表管理

在现有抄表功能基础上扩展物业抄表特性：

```sql
CREATE TABLE pms_meter_reading (
  reading_id    BIGINT PRIMARY KEY AUTO_INCREMENT,
  project_id    BIGINT NOT NULL,
  meter_id      BIGINT NOT NULL,
  period        VARCHAR(7) COMMENT '抄表期间，格式 2026-05',
  last_reading  DECIMAL(12,4) COMMENT '上次读数',
  curr_reading  DECIMAL(12,4) COMMENT '本次读数',
  usage_amount  DECIMAL(12,4) COMMENT '用量（自动计算：本次-上次）*倍率',
  loss_rate     DECIMAL(5,4) COMMENT '损耗比率',
  public_share  DECIMAL(12,4) COMMENT '公摊量',
  billed_amount DECIMAL(12,4) COMMENT '实际计费量（用量-损耗+分摊）',
  status        TINYINT COMMENT '1已录入 2已入账 3已复核',
  reader_id     BIGINT COMMENT '抄表员',
  read_time     DATETIME,
  import_batch  VARCHAR(50) COMMENT '批量导入批次号'
);
```

**功能要点实现**：

- **批量导入**：接收 Excel 文件（POI 解析），按 meter_code 匹配仪表，校验读数合理性（不低于上次读数），支持预警检测（用量超出阈值提示异常）
- **公摊设置**：总表用量 - 各分表用量之和 = 公摊量，按房间面积比例分摊
- **单户/批量入账**：入账时自动生成 `pms_receivable` 记录，状态流转：已录入 → 已入账

#### 3.4.5 生成应收

```sql
CREATE TABLE pms_receivable (
  recv_id       BIGINT PRIMARY KEY AUTO_INCREMENT,
  project_id    BIGINT NOT NULL,
  room_id       BIGINT NOT NULL,
  customer_id   BIGINT,
  fee_id        BIGINT NOT NULL,
  bill_period   VARCHAR(7) NOT NULL COMMENT '账单月 2026-05',
  amount        DECIMAL(12,2) NOT NULL COMMENT '应收金额',
  overdue_amt   DECIMAL(12,2) DEFAULT 0 COMMENT '滞纳金',
  recv_type     TINYINT COMMENT '1周期费 2临时费',
  status        TINYINT DEFAULT 0 COMMENT '0未收 1部分收 2已收 3减免',
  gen_time      DATETIME COMMENT '生成时间',
  gen_batch     VARCHAR(50) COMMENT '批次号，便于批量删除',
  memo          VARCHAR(500)
);
```

批量生成接口逻辑：

```
1. 按 project_id + bill_period 查询该月已生成记录（防重复）
2. 遍历所有有效费用分配（pms_fee_allocation）
3. 根据 calc_type 调用对应计算器（固定/面积/用量/公式）
4. 批量 INSERT pms_receivable（使用 MyBatis-Plus saveBatch，每批500条）
5. 记录生成批次号，支持按批次删除重新生成
```

**收款核心接口事务设计**（采用 `@Transactional`）：

```java
@Transactional(rollbackFor = Exception.class)
public PayResult collect(CollectRequest req) {
    // 1. 锁定应收记录（SELECT FOR UPDATE）
    // 2. 计算实收金额（含优惠/减免）
    // 3. 生成票据编号（Redis INCR）
    // 4. 更新 pms_receivable 状态
    // 5. 写入 pms_payment 收款记录
    // 6. 若有预收冲抵，更新 pms_prepayment
    // 7. 打印收据（异步，发 MQ 或直接返回打印数据）
    // 8. 清除看板缓存
}
```

## 4. C端微信端技术方案

### 4.1 整体架构

```
微信小程序/公众号（C端）
        │
        │  HTTPS API
        ▼
zhaoxinwy-wxmp（新增模块，独立端口8081）
        │
        ├── 微信授权登录（code → openid → JWT）
        ├── 业主绑定（openid ↔ pms_customer）
        └── 业务接口（复用 pms 服务层）
        │
        ▼
zhaoxinwy-pms（复用业务逻辑）+ MySQL + Redis
```

**部署建议**：wxmp 模块独立打包为独立 JAR，与 admin 同机部署，通过 Nginx 按路径分发（`/wxmp/` 转发至 8081 端口）。

### 4.2 微信授权登录与业主绑定

```
用户进入小程序
    │
    ▼
wx.login() 获取 code
    │
    ▼
POST /wxmp/auth/login { code }
    │
    ├── 调用微信 API 换取 openid + session_key
    ├── 查询 pms_customer WHERE openid = ?
    │       ├── 已绑定 → 返回 JWT token
    │       └── 未绑定 → 返回 needBind=true
    ▼
业主绑定页面：输入手机号 + 短信验证码
    │
    ▼
POST /wxmp/auth/bind { phone, smsCode, openid }
    │
    ├── 验证短信验证码
    ├── 查询 pms_customer WHERE phone = ?
    ├── 更新 openid 字段
    └── 返回 JWT token（payload含 customerId, projectId, roomIds）
```

## 6. 接口设计规范

### 6.1 统一响应格式

复用现有 `AjaxResult`：

```json
{
  "code": 200,
  "msg": "操作成功",
  "data": { ... }
}
```

### 6.2 分页规范

使用现有 `PageDomain` + MyBatis-Plus `IPage`，前端传 `pageNum`、`pageSize`。

### 6.3 C端接口鉴权

wxmp 模块使用独立 JWT，payload 包含：

```json
{
  "customerId": 1001,
  "openid": "oXxx...",
  "projectId": 5,
  "roomIds": [101, 102],
  "exp": 1748000000
}
```

Token 有效期7天，续签采用 Redis 存储 refreshToken（30天有效）。

### 6.4 关键接口幂等性

收款、退款、微信支付回调等核心接口使用 Redis 分布式锁保证幂等：

```java
// 以 orderId 为锁 key，防止回调重复处理
String lockKey = "pay:notify:" + orderId;
boolean locked = redisTemplate.opsForValue()
    .setIfAbsent(lockKey, "1", 60, TimeUnit.SECONDS);
if (!locked) return; // 已处理，直接返回
```

## 7. 安全与权限设计

### 7.1 B端权限扩展

在现有 RBAC（角色-菜单权限）基础上，增加**数据维度权限**（小区维度）：

```
权限层次：
  角色权限（菜单/操作）  → 控制能做什么
  小区权限（项目ID列表） → 控制能看哪个小区的数据
```

### 7.2 敏感数据处理

- 业主身份证号：AES-256 加密存储，查询时脱敏展示（`110***1234`）
- 业主手机号：展示时脱敏（`138****8888`），仅管理员可查看完整号码
- 微信支付回调：验签使用现有微信 SDK，密钥存储在配置文件（不进代码库）

### 7.3 操作日志

所有收款、退款、删除应收、调账操作通过 AOP 记录操作日志，写入 `sys_oper_log`（复用现有日志表）。

```yaml
wx:
  mp:
    appId: ${WX_MP_APPID}
    secret: ${WX_MP_SECRET}

    template:

    mchId: ${WX_PAY_MCHID}
```

## 9. 定时任务设计

新增以下 Quartz Job，注册至现有调度中心（`zhaoxinwy-quartz` 模块）：

| Job 类名 | Cron 表达式 | 功能 |
|---|---|---|
| `DashboardPushJob` | `0 0 18 * * ?` | 每日18:00推送管理层收费日报 |
| `OverdueCalcJob` | `0 0 1 * * ?` | 每日01:00计算并更新滞纳金 |
| `BillAutoGenJob` | `0 0 2 1 * ?` | 每月1日02:00批量生成当月应收 |
| `WxOrderExpireJob` | `0 */5 * * * ?` | 每5分钟关闭超时微信订单 |
| `PushRetryJob` | `0 0 */2 * * ?` | 每2小时重试失败的推送消息 |
| `ArrearsNoticeJob` | `0 0 9 10 * ?` | 每月10日09:00自动发送欠费提醒 |

## 10. 部署与扩展建议

### 10.1 新增服务部署

在现有 `docker-compose.yml` 中新增 wxmp 服务：

```yaml
zhaoxinwy-wxmp:
  image: openjdk:8-jre-buster
  volumes:
    - ./zhaoxinwy-wxmp.jar:/app/app.jar
    - /data/upload:/data/upload
  ports:
    - "8081:8081"
  environment:
    - SPRING_PROFILES_ACTIVE=prod
  depends_on:
    - mysql
    - redis
```

Nginx 新增路由：

```nginx
# 微信端 API
location /wxmp/ {
    proxy_pass http://zhaoxinwy-wxmp:8081/wxmp/;
}

# 微信小程序静态资源（如使用 uniapp 编译）
location /miniapp/ {
    root /data/dist;
    try_files $uri $uri/ /miniapp/index.html;
}
```

### 10.2 性能优化建议

- 看板数据使用 Redis 缓存，收费操作后主动刷新，避免每次实时聚合
- 批量生成应收（月初高并发）使用异步任务 + 进度轮询，前端展示进度条
- 报表查询超过1万条时，强制要求传入时间范围筛选条件
- 微信推送使用队列化（Redis List 模拟简单队列），控制发送频率符合微信接口限制

### 10.3 数据迁移建议

若现有商业地产数据需迁移至住宅物业体系，提供数据迁移脚本：商铺 → 房间，商铺业主 → 客户档案，保留原有收费历史数据。

## 11. 开发排期建议

### 11.1 优先级划分

**P0（核心收费链路，优先交付）**：

- 基础档案（房间/客户/仪表/车位）
- 费用定义 & 费用分配
- 生成应收 & 收银台
- 抄表管理
- 数据查询报表（Top 5：交易汇总/明细/收费率/欠费明细/应收管理）

**P1（管理增强，第二阶段）**：

- 收费模块看板 & 定时推送
- 企业信息（公司/组织/权限）
- 完整报表体系（剩余报表）
- 票据管理 & 发票换开

**P2（C端微信，第三阶段）**：

- 微信端授权登录 & 业主绑定
- 微信缴费 & 电子收据
- 通知公告 & 欠费通知推送
- 管家展示 & 便民信息
- 信息精准推送
