## 1. 现状分析与设计策略

### 1.1 现有收费体系的结构与局限

通过分析 `pms_schema.sql` 和 `system-overview.md`，现有收费体系的核心表链路为：

```
config_house_block（商圈）
  └── config_house（商铺）
        └── payment_contract（合同）
              ├── payment_contract_fee（合同收费项）
              ├── payment_bill（账单，含抄表数据内嵌）
              ├── payment_deposit（押金）
              └── payment_pre（预收款）
                    └── payment_pre_account（预收款余额账户）
payment_pay_log（流水，跨所有收费类型）
payment_order（在线支付订单，对接微信/支付宝）
```

**对应的已有 API**（`zhaoxinwy-pms` 模块，路径均以 `/payment/` 或 `/baseconfig/` 开头）：

| 功能 | 路径 | 状态 |
|---|---|---|
| 商铺/商圈管理 | `/baseconfig/House`, `/baseconfig/ConfigHouseBlock` | ✅ 可用 |
| 合同管理 | `/payment/PaymentContract` | ✅ 可用 |
| 账单生成 | `/payment/PaymentBillCreate` | ✅ 可用 |
| 收款/退款 | `/payment/PaymentBillPay` | ✅ 可用 |
| 抄表管理 | `/payment/PaymentMeter` | ✅ 功能较弱 |
| 收费项配置 | `/baseconfig/ConfigFeeItem` | ✅ 字段不足 |
| 统计报表 | `/statistics/DailyReport` 等 | ✅ 功能有限 |

**局限性**：

- 资产模型绑定商业地产（商圈→商铺），无法直接支撑住宅物业（项目→楼栋→单元→房间→业主）的层级档案
- `payment_bill` 中抄表数据（`last_index`, `current_index`）内嵌，无独立抄表期间管理，不支持公摊、损耗批量核算
- `config_fee_item.formula` 字段（varchar 50）过短，不支持复杂梯度公式
- 缺少票据编码管理、票据模板自定义、发票换开等物业收费专用功能
- 缺少收银台概念（按房间查费、专款专冲、批量生成/删除应收等操作入口）
- 收缴率、欠费明细、日结等 20+ 维度专业报表缺失
- `owner_user`, `owner_wx_user` 表存在但 `zhaoxinwy-owner` 模块源码缺失，C端能力需重建

### 1.2 设计策略

**核心原则：建立独立的住宅物业收费子系统，与现有商业收费体系并行，共用基础框架（用户/角色/权限/日志/定时任务），不修改现有任何表结构与接口。**

| 维度 | 决策 |
|---|---|
| 后端归属 | 在 `zhaoxinwy-pms` 模块内新建包 `com.zhaoxinms.resi`（residential），与现有 `payment/baseconfig` 包并列 |
| C端归属 | 新建 `zhaoxinwy-wxmp` 模块，复用 `owner_wx_user` 表结构（扩展字段），重写 API |
| 表命名 | 全部以 `resi_` 前缀，与现有 `payment_/config_` 前缀区隔 |
| 接口路径 | 全部以 `/resi/` 开头，与现有 `/payment/` 完全隔离 |
| 票据流水号 | 复用现有 `base_billrule` + `BillRuleController`（`/Base/BillRule`），新增住宅收费专用单据规则记录 |
| 消息推送 | 复用现有 `sys_sms` + `sys_sms_template` 体系 |
| 操作日志 | 复用现有 `sys_oper_log`（AOP 自动记录） |
| 定时任务 | 在现有 `sys_job`（Quartz）中新增任务记录 |

### 2.2 数据流总览

```
【档案层】
  resi_project → resi_building → resi_room ← resi_customer（绑定）
                                           ← resi_meter_device（挂表）
                                           ← resi_parking_space（车位）

【配置层】
  resi_fee_definition（费用定义）
    └── resi_fee_allocation（分配到房间）

【计费层】
  resi_meter_reading（抄表）→ 触发生成 resi_receivable（应收账单）
  resi_receivable_gen（批量生成周期费）→ resi_receivable

【收款层】
  resi_cashier（收银台操作）→ resi_pay_log（流水）
                            → 更新 resi_receivable.pay_state
                            → 扣减 resi_pre_account（预收款余额）
                            → 生成 resi_receipt（收据）

【查询层】
  resi_pay_log + resi_receivable → 各维度报表接口
```

## 3. 数据库设计（全量DDL）

> 字符集统一 `utf8mb4`，主键策略：档案类用 `BIGINT AUTO_INCREMENT`，流水类用 `VARCHAR(32)` UUID（与现有 `payment_pay_log.id` 保持一致）。

### 3.1 基础档案

```sql
-- 项目（小区）管理
CREATE TABLE `resi_project` (
  `id`            BIGINT NOT NULL AUTO_INCREMENT,
  `code`          VARCHAR(50)  NOT NULL COMMENT '项目编号',
  `name`          VARCHAR(100) NOT NULL COMMENT '项目名称',
  `address`       VARCHAR(255) COMMENT '详细地址',
  `contact_phone` VARCHAR(20)  COMMENT '联系电话',
  `manager_name`  VARCHAR(50)  COMMENT '负责人姓名',
  `manager_phone` VARCHAR(20)  COMMENT '负责人电话',
  `logo_url`      VARCHAR(500) COMMENT '项目Logo',
  `seal_url`      VARCHAR(500) COMMENT '公章图片（收据打印用）',
  `enabled_mark`  TINYINT NOT NULL DEFAULT 1,
  `create_by`     VARCHAR(64) DEFAULT '',
  `create_time`   DATETIME,
  `update_by`     VARCHAR(64) DEFAULT '',
  `update_time`   DATETIME,
  PRIMARY KEY (`id`),
  UNIQUE KEY `uk_project_code` (`code`)

  `floors`      INT COMMENT '总楼层数',
  `enabled_mark` TINYINT NOT NULL DEFAULT 1,
  `create_by`   VARCHAR(64) DEFAULT '',
  `create_time` DATETIME,
  `update_by`   VARCHAR(64) DEFAULT '',
  `update_time` DATETIME,
  PRIMARY KEY (`id`),
  KEY `idx_project_id` (`project_id`)

-- 房间管理（住宅物业核心档案）
CREATE TABLE `resi_room` (
  `id`               BIGINT NOT NULL AUTO_INCREMENT,
  `project_id`       BIGINT NOT NULL,
  `building_id`      BIGINT NOT NULL,
  `unit_no`          VARCHAR(20) COMMENT '单元号',
  `floor_no`         INT         COMMENT '楼层',
  `room_no`          VARCHAR(50) NOT NULL COMMENT '房号，如101',

  `building_area`    DECIMAL(10,2) COMMENT '建筑面积(㎡)',
  `inner_area`       DECIMAL(10,2) COMMENT '套内面积(㎡)',

  `garage_no`        VARCHAR(50)  COMMENT '附属车库编号',
  `storage_no`       VARCHAR(50)  COMMENT '附属储藏室编号',
  `state`            VARCHAR(20) NOT NULL DEFAULT 'NORMAL' COMMENT 'NORMAL/VACANT/DECORATING/TRANSFERRED',
  `remark`           VARCHAR(500),
  `enabled_mark`     TINYINT NOT NULL DEFAULT 1,
  `create_by`        VARCHAR(64) DEFAULT '',
  `create_time`      DATETIME,
  `update_by`        VARCHAR(64) DEFAULT '',
  `update_time`      DATETIME,
  PRIMARY KEY (`id`),
  KEY `idx_project_building` (`project_id`, `building_id`),
  UNIQUE KEY `uk_room` (`project_id`, `building_id`, `unit_no`, `room_no`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='房间档案';

-- 客户（业主/租户）管理
CREATE TABLE `resi_customer` (
  `id`            BIGINT NOT NULL AUTO_INCREMENT,
  `project_id`    BIGINT NOT NULL,
  `customer_name` VARCHAR(100) NOT NULL,
  `phone`         VARCHAR(20)  NOT NULL,
  `id_card`       VARCHAR(64)  COMMENT '身份证号（AES加密存储）',
  `gender`        TINYINT      COMMENT '0未知 1男 2女',
  `customer_type` TINYINT DEFAULT 1 COMMENT '1业主 2租户 3临时',
  `openid`        VARCHAR(128) COMMENT '微信openid（绑定后填入）',
  `enabled_mark`  TINYINT NOT NULL DEFAULT 1,
  `create_by`     VARCHAR(64) DEFAULT '',
  `create_time`   DATETIME,
  `update_by`     VARCHAR(64) DEFAULT '',

-- 客户与资产绑定关系
CREATE TABLE `resi_customer_asset` (
  `id`          BIGINT NOT NULL AUTO_INCREMENT,

  `asset_id`    BIGINT NOT NULL,
  `bind_date`   DATE   NOT NULL,
  `unbind_date` DATE   COMMENT 'NULL=当前有效',

-- 仪表档案
```

```sql
  `invoice_title`  VARCHAR(200)  COMMENT '发票项目名称',
  `tax_rate`       DECIMAL(5,2)  COMMENT '税率%',
  `sort_code`      INT DEFAULT 0,
  `enabled_mark`   TINYINT NOT NULL DEFAULT 1,
  `creator_time`   DATETIME,
  `creator_user_id` VARCHAR(50),
  `last_modify_time` DATETIME,
  `last_modify_user_id` VARCHAR(50),
```

## 4. 后端模块实现方案

> 所有 Controller 继承现有 `BaseController`，使用现有 `AjaxResult` 统一响应，分页用 `PageDomain` + MyBatis-Plus `IPage`，操作日志由现有 AOP 自动记录到 `sys_oper_log`。

### 4.1 基础档案

**接口路径规则**：`/resi/archive/{实体名}`

关键实现要点：

**项目权限隔离**：在 `SecurityUtils.getUserId()` 基础上，通过用户与项目关联控制数据范围。新建 `ResiProjectScope` 注解 + AOP，自动将查询条件追加 `project_id IN (?)` 过滤，与现有角色数据权限（`sys_role.data_scope`）并行工作。

**房间搜索（收银台入口）**：提供模糊搜索接口，按 `room_alias`、`customer_name`、`room_no` 联合模糊匹配，返回树形结构（项目→楼栋→房间），支持下钻查询。

**房屋过户**：
```
POST /resi/archive/room/transfer
事务操作：
  1. 终止旧业主绑定（resi_customer_asset.is_current=0, unbind_date=today）
  2. 新增新业主绑定（resi_customer_asset）
  3. 记录过户日志（resi_room_transfer）
  4. 更新房间冗余字段（冗余客户姓名若有）
```

### 4.2 费用定义与分配

**公式引擎**：`resi_fee_definition.formula` 字段为 TEXT，直接复用现有 `config_fee_item` 的公式引擎（`if/elsif/else/return` 语法，变量：`单价`、`数量`），在 Service 层注入公式计算器 Bean。

**批量费用分配**：
```
POST /resi/feeconfig/allocation/batch
参数：fee_id + 批量方式（按楼栋/按单元/按项目全量）+ 生效日期
逻辑：查询符合条件的所有 resi_room，批量 INSERT resi_fee_allocation（MyBatis-Plus saveBatch，500条一批）
```

### 4.3 抄表管理

**Excel 批量导入**：复用现有 `PaymentMeterImportController` 模式（POI 解析），新建 `ResiMeterImportController`：

```
GET  /resi/meter/reading/template         # 下载导入模板
POST /resi/meter/reading/import/upload    # 上传 Excel
GET  /resi/meter/reading/import/preview  # 预览数据+异常行提示
POST /resi/meter/reading/import/confirm  # 确认导入，写入resi_meter_reading
```

**公摊计算逻辑**：
```
同一 public_group 的总表用量 - 各分户表用量之和 = 公摊总量
各分户分摊量 = 公摊总量 × (该户建筑面积 / 组内所有户建筑面积之和)
写入每条 resi_meter_reading.share_amount
```

**预警检测**：抄表导入时检查 `curr_reading < last_reading`（读数回退）或 `(curr-last)*multiplier > threshold`（用量异常），标记警告行，不阻断导入但在列表中高亮显示。

**入账**：单户或批量入账时，调用 `ResiReceivableService.createFromMeterReading()`，生成 `resi_receivable` 记录，同时更新 `resi_meter_reading.status=BILLED` 和 `receivable_id`。

### 4.4 应收生成

**周期费批量生成**：

```java
// ResiReceivableGenService.java
@Transactional(rollbackFor = Exception.class)
public GenResult batchGenerate(Long projectId, String billPeriod) {
    // 1. 防重复：查询该月是否已生成（按project_id+bill_period聚合）
    // 2. 生成批次号：genBatch = "GEN-" + projectId + "-" + billPeriod
    // 3. 查询所有有效费用分配（resi_fee_allocation，start_date<=today，end_date IS NULL OR end_date>=today）
    // 4. 按 fee_type=PERIOD 过滤，遍历每条分配记录
    // 5. 根据 calc_type 调用对应计算器（固定价/面积*单价/公式引擎）
    // 6. 计算滞纳金（若上月有欠费）
    // 7. 批量INSERT resi_receivable（每批500条，MyBatis-Plus saveBatch）
    // 8. 记录生成批次，返回生成结果（总数、成功数、跳过数）
}
```

**支持按月重新生成**：先删除该批次下状态为"未收"的应收记录，再重新生成。已收费用不可删除，会在重新生成时跳过（UNIQUE KEY 约束）。

**临时费**：

```
POST /resi/receivable/create-temp
参数：resource_id, fee_id, num, price（可覆盖），remark
说明：直接插入单条 resi_receivable（fee_type=TEMP），不走批量生成流程
```

### 4.5 收银台

收银台是整个系统操作频率最高的入口，接口设计如下：

```
# 查询入口
GET  /resi/cashier/room/search?keyword=&projectId=    # 模糊搜索房间
GET  /resi/cashier/room/{roomId}/receivables          # 查询房间待缴费用
GET  /resi/cashier/room/{roomId}/summary              # 应收/已缴/欠费汇总

# 费用筛选参数（支持：按费用名/年份/账单月/费用类型）
GET  /resi/cashier/room/{roomId}/receivables?feeId=&year=&period=&feeType=

# 收款核心
POST /resi/cashier/collect                            # 收款（含预收冲抵）
POST /resi/cashier/refund                             # 退款
```

**收款核心接口事务设计**：

```java
@Transactional(rollbackFor = Exception.class)
public CollectResult collect(CollectRequest req) {
    // 1. 参数校验：receivableIds 不为空，payAmount > 0
    // 2. SELECT FOR UPDATE 锁定应收记录（防并发重复收款）
    //    QueryWrapper.eq("pay_state","0").in("id", req.getReceivableIds()).last("FOR UPDATE")
    // 3. 计算本次应收合计（total + overdue_fee - discount）
    // 4. 生成收据号（复用 base_billrule，调用 /Base/BillRule/useBillNumber/{enCode}）
    //    或者：Redis INCR + 格式化，key = "receipt_no:{projectId}:{yyyyMMdd}"
    // 5. 若有预收冲抵：
    //    a. 查询 resi_pre_account 余额 FOR UPDATE
    //    b. 校验余额充足（非专款专冲）或 fee_id 匹配（专款专冲）
    //    c. 扣减余额，写入 resi_pre_pay（op_type=OUT）
    // 6. 批量更新 resi_receivable（pay_state, paid_amount, pay_log_id, pay_time）
    // 7. INSERT resi_pay_log（本次收款流水）
    // 8. 若有押金类费用：INSERT resi_deposit
    // 9. 清除看板 Redis 缓存（key: dashboard:{projectId}:*）
    // 10. 返回收据数据（供前端调起打印）
}
```

**票据号生成**：优先复用 `base_billrule` 的流水规则（现有 `/Base/BillRule/useBillNumber/{enCode}` 接口，已实现数据库级别的原子自增，见 `BillRuleController` 第197行）。为住宅收费新增一条规则记录，如前缀 `ZS`、日期格式 `yyyyMMdd`、流水位数6位，示例：`ZS202605100001`。

### 4.7 折扣管理

折扣配置由 B 端维护，收银台操作时可手动选择应用某条折扣，或由系统根据有效期自动推荐。折扣仅在收款时影响 `receivable`（应收合计），不回写原始 `total`，保留明细追溯。

### 4.8 预收款与押金

## 6. C端微信端方案

### 6.1 模块定位

`zhaoxinwy-wxmp` 为独立 Spring Boot 应用，独立端口（8081），通过内部 HTTP 调用或直接引用 `zhaoxinwy-pms` 的 Service 层（推荐同进程内调用，简化部署）。

现有 `owner_wx_user` 表有结构，但 `zhaoxinwy-owner` 源码缺失。本方案新建 `resi_wx_user` 表，自行实现 C 端登录绑定逻辑。

### 6.2 微信授权与业主绑定

```
小程序 wx.login() → code
  ↓
POST /wxmp/auth/login { code, appType }
  ↓
后端：微信 code2session API → openid
  ↓
查询 resi_wx_user WHERE openid=?
  ├── 已绑定客户 → 生成 JWT（payload: customerId, projectId, roomIds）→ 返回
  └── 未绑定 → 返回 { needBind: true, openid }
  ↓
业主输入手机号 + 短信验证码（通过 sys_sms 发送）
  ↓
POST /wxmp/auth/bind { phone, smsCode, openid }
  ↓
匹配 resi_customer.phone → 写入 openid → INSERT resi_wx_user → 返回 JWT
```

### 6.3 微信缴费核心流程

```
C端查询欠费：GET /wxmp/payment/bills
  ↓（返回 resi_receivable 欠费列表）
业主勾选费用 → POST /wxmp/payment/pre-create { recvIds[] }
  ↓
  后端：汇总金额 → 查重（防重复下单）→ 调微信统一下单API
  → 返回 prepayId
  ↓
小程序唤起微信支付
  ↓
微信支付成功回调：POST /wxmp/payment/notify（微信服务器 → 后端）
  ↓
  验签 → 幂等校验（Redis setIfAbsent）→ 调 ResiCashierService.collect()
  → 更新 resi_receivable → 写 resi_pay_log（client=2）
  → 推送支付成功微信模板消息
  ↓
C端查看电子收据：GET /wxmp/payment/receipt/{payLogId}
```

### 6.4 欠费通知推送

```
B端：POST /resi/notice/send-arrears { projectId, billPeriod, roomIds? }
  ↓
后端：查询欠费业主及 openid → 写入 resi_push_record（status=0）
  ↓
Quartz 扫描（每5分钟）→ 取待发记录 → 调微信模板消息API（限速：200条/秒）
→ 更新 resi_push_record.status
```

```javascript
  const cell = row.cells.find(c => c.period === column.property);
  if (!cell) return '';
  return { PAID: 'cell-green', UNPAID: 'cell-yellow', PART_PAID: 'cell-red', NOT_GEN: 'cell-gray' }[cell.status] || '';
}
```

## 11. 开发排期建议

### 11.1 优先级与阶段划分

**P0 — 核心收费链路（优先交付，约6～8周）**

- 费用定义 + 费用分配（含公式引擎对接）
- 抄表管理（含 Excel 导入、公摊计算）
- 周期费生成应收 + 临时费
- 收银台（收款/退款/冲红/预收款冲抵）

- 票据打印（收款单/缴费通知单）

**P1 — 管理增强（约4～5周）**

- 收费看板 + 定时推送
- 完整报表体系（20+报表全部）

- 便民信息

**测试与上线（约2～3周）**：联调、UAT、性能压测、部署。
